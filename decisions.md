### 実装
- **H1再設計**:旧「あなたの / 恋愛タイプと / 運命の10曲。」→「あなたの / 恋愛タイプから / ラブソングを診断。」→「あなたを / 16の恋愛タイプから / ラブソングを診断。」→ 最終 **「あなたを / 16の恋愛タイプから / ラブソング診断。」**。最後の「を」を削って **「ラブソング診断。」を名詞化**(サービス名「ラブソング診断16」と整合・締まり良)。「16の恋愛タイプ」「ラブソング」の両ワードに `.grad`(ピンク→コーラル→ゴールド)。タイトル単独で「16タイプ分類 × ラブソング診断」のサービス本質が伝わる構造
- **タイポ調整**:旧 `font-size: clamp(36px,11.5vw,46px)` → **`clamp(32px,9.5vw,42px)`**、`letter-spacing -.03em`、`line-height 1.22`、`word-break: keep-all` で禁則崩れ防止。文字数増えた分だけサイズを少し下げてバランス維持
- **リード文を削除**:旧「20問で、16タイプの中のあなたを当てる。あなたにぴったりのラブソングを10曲レコメンド。」も ファウンダー「タイトルと同じこと言い直してる、おかしい」指摘で全削除。タイトルが既に機能訴求を兼ねるため、メタピル4種(20問・3分 / 全16タイプ / 10曲プレイリスト / 無料・登録不要)で補強する構成に。`.hero-meta` の `margin: 26px 0 28px` でタイトルとの間隔調整
- **メタ表記再再設計(最終)**:旧4ピル「20問・3分 / 全16タイプ / 10曲プレイリスト / 無料・登録不要」→ 3ピル数字強調「3分 / 16タイプ / 10曲」→ 最終 **「#16タイプ #所要時間3分 #完全無料」**(SNSハッシュタグ風・ピル形廃止・素テキスト)。ファウンダー「もっとスッキリ」→ 最終「ハッシュタグ風」指定で確定。ピル形より素テキストの方がSNS文脈と整合・ヒーローの視覚密度が一気に軽くなる。ピンク色テキスト1行(`gap: 14px`、`font-size: 13.5px`)、`.hm-n`(数字強調用)は不要になり廃止
- **マーキュー再設計**:旧 大型テキスト(19px・♪装飾) → **白背景ピル + タイプ色ドット + タイプ色文字(50%濃度)+ タイプ色ボーダー + タイプ色影**。16タイプの色彩世界観を一覧で予告
- **「↓ こんな結果が出る」ラベル**:マーキュー上に小キャプション追加(`.hp-label`)。スクロール誘導 + 期待値設計
- `app/css/style.css`:`.hero-lead b`(マーカー強調)/ `.hero-preview` `.hp-label`(プレビュー枠)/ `.marquee` `.mq` `.mq-dot`(ピル化)を全面改修

- **結果(ショウリールの主役)**:ScrollTrigger でパネルごとに reveal をスタッガー表示、**タイプ名を1文字ずつリビール**、入場時に**紙吹雪**、「10」の**カウントアップ**、モチーフの視差スクロール。
- **質感**:CTAのグロー、qcardのグラデ、円の影など premium 寄りに。

#### 注意 / メモ
- 結果リビールは GSAP時=ScrollTrigger、未読込時=IntersectionObserver(`observeReveals`)のフォールバックに自動切替。
- CSSはブラウザキャッシュで旧版が残る事象があったため、`index.html` の CSS リンクに `?v=` を付与(開発時の確実な反映用)。
- 外部スクリプトは現状 SRI 無し(プロトのため)。本番では `integrity`/`crossorigin` 付与 or セルフホスト推奨。
- キャラ絵は引き続き未(タイプ色カード+モーションで代替)。本番イラスト差し替えが次の大きな一手。

検証:Playwrightで GSAP読込・ヒーロー入場・行マスク・マーキー・ワイプ遷移・スケール選択の弾け・結果のScrollTrigger reveal/char/カウントアップ・紙吹雪・console 0 error を確認。

#### 不具合修正:スケールの円が選択できない(回答不能)
- 症状:「そう思う/思わない」の円が反応しない。原因は **クイズ入場の GSAP タイムラインが progress 0 で停止 → `from` の初期値 `scale(0)` だけが残り、円の当たり判定がゼロ**になっていた(GSAPタイムラインが何らかの理由で再生されず固着)。
- 対策:**最重要操作であるクイズ入場を GSAP から CSSアニメーション(@keyframes + animation-fill-mode:backwards)に変更**。CSSアニメは必ず最終状態(scale1=クリック可)に到達するため固着しない。GSAPはクリック時の弾け演出(dotBurst)のみ担当。
- 教訓:**インタラクティブ要素の入場に `gsap.from`(初期状態を要素に焼き付ける方式)を使わない**。停止時に操作不能化するため、CSSアニメか、停止しても操作可能な方式にする。
- 併せて CSS/JS のキャッシュ対策で `?v=4` に更新。

#### 3層構造
1. **Layer 1: 楽曲ソース** = Spotify / Apple Music API から実在楽曲を取得(5,000〜10,000曲規模)

3. **Layer 3: 軽い人間監修** = 音楽酸素。さんが「明らかにおかしい」だけNGする画面で品質チェック

ユーザーが (タイプ, ジャンル) を確定 → スコア閾値超え曲のプールから**ランダム10曲抽出** → 「別の10曲」ボタンで何度でも引き直し可

- 日次/週次の集計バッチでスコア更新

### 楽曲レコメンドのアーキテクチャ大幅改定:Type × Genre × ランダム

**変更前**:16タイプ → 各10曲固定(計160曲)
**変更後**:16タイプ × 8ジャンル のマトリックスで、各セルに **プール30〜50曲**、毎回 **ランダム10曲抽出**

#### Why
- 普段聞くジャンルから外れた曲をレコメンドしてもシェアされない(K-POP好きにJ-POP出しても刺さらない)
- 同じタイプのAさんとBさんに違う曲を出すことで**シェア再投稿の動機**が生まれる(「私も同じタイプだけど違う曲!」)
- TikTokで複数人が同じ結果を投稿しても**曲がバラけて発見感**が出る
- 編集側で季節曲・新譜の差し替えが容易になる
- 1回診断したユーザーの**リピート再診断**を促進

#### データモデル
楽曲に多次元タグを付けて DB 化:
- `genres[]`(複数ジャンル可)
- `types[]`(複数タイプ可) ← 1曲が複数タイプにマッチしてOK
- `weight`(ピン留め・編集側の重み付け用)

#### 結果ページに「別の10曲も見る」ボタンをデフォルトON
再回遊・再シェアを促進する目的。実装必須機能として確定。

#### 編集ボリューム目標
- **ローンチ最低ライン**:各セル20曲 = 計1,000曲規模(重複考慮で実数はもっと少)
- **ストレッチ目標**:各セル50曲 = 計2,000曲規模(重複考慮)

- 新:**ジャンル選択**(マルチ・最大3つ予定) → 質問12〜15問 → タイプ判定 → 楽曲提示

### 入力設計最終確定:**ジャンル選択(1〜5) + 3アーティスト入力 の両方を採用**

ファウンダー指示「両方入れてもらう」を受けて、当初の「ジャンル選択 vs アーティスト入力 どちらかメイン」議論を**両軸採用**に集約。

#### フロー
- Step 0a:ジャンル選択(1〜5個 / 18ジャンルから / 「ジャンルこだわらない」もOK)
- Step 0b:好きなアーティスト3組(Spotifyオートコンプリート / 「思いつかない」Skip可)
- Step 1:タイプ判定 12〜15問
- Step 2:タイプ結果
- Step 3:10曲レコメンド(3軸スコア合成)

#### 推薦の3軸スコア合成
- `final = type_score × 0.5 + genre_match × 0.25 + artist_similarity × 0.25`
- タイプ適合度を主軸に、ジャンル/アーティスト軸で精緻化

#### なぜ両方やるか
- **ジャンル単独**:浅い、アーティスト名まで踏み込めない
- **アーティスト単独**:思いつかない人が離脱
- **両方**:取りこぼし防止 + 推薦精度爆上げ + 結果カードに2軸両方表示できる(シェア性最強)

#### 結果カード表示例
> NewJeans / ヨルシカ / 藤井風 好きの **沼っくま**

### 旧:「3アーティスト入力」をメインに採用案(没・両方採用に統合)

最終アーキテクチャ確定。ジャンル選択方式から **3アーティスト入力方式** に大幅刷新。

#### Why
- TikTok層は **ジャンル単位ではなくアーティスト単位で自己同一化**(Z世代:「Kpop民」より「NewJeans/IVE/aespa が好き」が自然)
- アーティスト入力のほうが**vibe・歌詞・年代まで含めた個人特性を一発で捕捉**できる
- 結果カードに「好きなアーティスト:◯/◯/◯ → タイプ:沼っくま」を載せれば**スクショ映え・シェア性爆発**
- ジャンルタクソノミー定義の労力が消える(Spotifyカタログ全体を流用)
- AI計算が事前マトリックスじゃなく**ユーザー入力ごとの動的計算+キャッシュ**になり、無駄なAIコールが消える

#### 技術フロー
1. ユーザー入力 3アーティスト(Spotify Search でオートコンプリート)
2. Get Related Artists API で各アーティストの関連アーティスト取得
3. Top Tracks API で代表曲取得 → 候補プール 300〜500曲
4. LLM が各曲を診断結果タイプとの適合度スコア付与(**キャッシュ化:同じ曲は再判定しない**)
5. スコア閾値超え曲群 → ランダム10曲抽出
6. 「別の10曲」ボタンで再抽選可

#### フォールバック
「好きなアーティストが思いつかない」ユーザー向けに **「ジャンルから選ぶ」リンク** を併置。18ジャンル選択モードに遷移。

#### 18ジャンル(フォールバック用、保持)
- 邦楽メジャー(6):J-POP / J-ROCK / J-オルタナ・J-インディー / シティポップ / J-HIPHOP / J-R&B・J-ソウル
- 邦アイドル(5):坂道系 / ジャニーズ系 / 48系・その他 / LDH系 / V系・ヴィジュアル系
- アニメ・サブカル(2):アニソン / ボカロ
- 韓国(3):K-POP(ガールズ) / K-POP(ボーイズ) / K-HIPHOP・K-R&B
- 洋楽(2):洋楽POP・R&B / 洋オルタナ・インディー

#### 注意点
- **Spotify Recommendations API は2024年11月から新規アプリ制限**:Related Artists / Top Tracks / Search の組み合わせで自前ロジック構築。問題なし
- **オートコンプリート実装必須**:タイプミス・表記揺れ(Sakanaction vs サカナクション)を吸収するため

## 2026-05-20(命名フェーズ詳細)

### #5 ヤキモチ型:「進撃のヤキモチモンスター」→「ヤキモチモンスター」
「進撃の◯」枕詞をレアアイテム化するため、#3 進撃のロマンチストだけに残し短縮。

### #11 ミステリアス型:「ミステリーキャット」却下 → 「ミステリアス狼」採用
理由:(1)「ミステリー」が名詞でミステリアスとズレ、(2)『キャッツ』Macavity 固有名詞被り。狼=ローンウルフ / 月夜 / 単独行動 でミステリアス型と完全合致。カタカナ「ミステリアス」+ 漢字「狼」のミックスが可愛さを足す。

### #12 直球型:「ド直球ガエル」検討 → 「ド直球ザウルス」採用
ガエル(ピョン!軽快)よりザウルス(ドスドス突進)のほうが直球型の本気度(=積極的・自分から動く)に直撃。中性的でプロフ貼り適性も高い。

### #14 情熱型:「暴走情熱ゾンビ」→「情熱ラブゾンビ」
元案は「暴走」と「情熱」が温度系で意味重複 + ゾンビ単体のネガ含意。「ラブ」を緩衝材に入れることでゾンビ感がコメディ寄りに(『ウォーム・ボディーズ』系の "愛で正気保つゾンビ" コメディ枠と接続)。

### #15 沼り型:「沼+動物」テンプレを経て「沼っくま」採用
最終的に造語「沼っくま」(沼った+くま)に決着。TikTok層の造語スラング(マブダチエイリアン系)感を取りに行った。

### コンセプト:「パーソナル恋愛ソング診断」
クライアント=音楽メディアであり、診断結果として楽曲をレコメンドする形が本業との親和性高 + TikTokシェア性高。

### グリッド方式(2軸×4)→ 独立16アーキタイプ方式に変更
当初2軸×4=16グリッドで検討したが、空きマス埋めのために不自然な組み合わせ(例:衝動入り×クール)を作る必要が出て破綻。独立16なら出したい候補を素直に16選べる。

**結果**:診断ロジックは「各タイプにスコアを割り振り、合計点が最高のタイプを返す」方式(軸ベース判定ではない)。
